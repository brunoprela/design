# Alembic Migrations

## Context & Problem

Database schemas change as the application evolves. Without a migration tool, schema changes are applied manually — leading to drift between environments, unreproducible deploys, and "it works on my machine" bugs.

Alembic provides versioned, sequential, reversible database migrations that are checked into source control alongside the code they support. Every environment runs the same migrations in the same order, producing identical schemas.

## Design Decisions

### One Migration History Per Bounded Context

In a modular monolith, each bounded context owns its own PostgreSQL schema. Alembic supports multiple migration directories via `--name` or separate `alembic.ini` sections. This prevents one module's migration from accidentally affecting another's tables.

```
migrations/
├── market_data/
│   ├── env.py
│   └── versions/
│       ├── 001_create_prices_table.py
│       └── 002_add_hypertable.py
├── positions/
│   ├── env.py
│   └── versions/
│       ├── 001_create_trades_table.py
│       └── 002_add_portfolio_index.py
└── alembic.ini
```

### Auto-Generate Then Review

Use `alembic revision --autogenerate` to detect model changes and generate a migration draft. Always review and edit the generated migration — autogenerate misses:

- Data migrations (backfilling new columns)
- Index creation strategy (`CONCURRENTLY` for zero-downtime)
- Custom SQL (functions, triggers, partitioning)
- Column renames (detected as drop + add)

### Migration Naming

Use descriptive names with a sequence prefix:

```
001_create_trades_table.py
002_add_portfolio_index.py
003_add_settlement_date_column.py
```

Avoid autogenerated hashes as the sole identifier — they are meaningless during review.

### env.py (Async)

```python
import asyncio
from logging.config import fileConfig

from alembic import context
from sqlalchemy.ext.asyncio import create_async_engine

from app.modules.positions.models import Base

config = context.config
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata


def run_migrations_offline() -> None:
    url = config.get_main_option("sqlalchemy.url")
    context.configure(url=url, target_metadata=target_metadata, literal_binds=True)
    with context.begin_transaction():
        context.run_migrations()


def do_run_migrations(connection):
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()


async def run_migrations_online() -> None:
    engine = create_async_engine(config.get_main_option("sqlalchemy.url"))
    async with engine.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await engine.dispose()


if context.is_offline_mode():
    run_migrations_offline()
else:
    asyncio.run(run_migrations_online())
```

### Example Migration

```python
"""Add settlement_date to trades table.

Revision ID: 003
Revises: 002
Create Date: 2025-03-15 14:30:00
"""

from alembic import op
import sqlalchemy as sa

revision = "003"
down_revision = "002"


def upgrade() -> None:
    # Add column as nullable first
    op.add_column(
        "trades",
        sa.Column("settlement_date", sa.Date(), nullable=True),
        schema="positions",
    )

    # Backfill: settlement_date = executed_at + 1 business day (simplified)
    op.execute("""
        UPDATE positions.trades
        SET settlement_date = (executed_at::date + INTERVAL '1 day')
        WHERE settlement_date IS NULL
    """)

    # Now make it non-nullable
    op.alter_column(
        "trades",
        "settlement_date",
        nullable=False,
        schema="positions",
    )


def downgrade() -> None:
    op.drop_column("trades", "settlement_date", schema="positions")
```

### Zero-Downtime Index Creation

```python
def upgrade() -> None:
    # CREATE INDEX CONCURRENTLY cannot run inside a transaction
    op.execute("""
        CREATE INDEX CONCURRENTLY IF NOT EXISTS
        ix_trades_settlement_date
        ON positions.trades (settlement_date)
    """)
```

Note: `CONCURRENTLY` requires the migration to run outside a transaction block. Configure this in the migration with `autocommit=True` or by splitting the migration.

## Migration Safety Checklist

Before merging any migration:

1. **Reversible** — does `downgrade()` actually undo the change?
2. **Idempotent** — can it run twice without error? (use `IF NOT EXISTS`, `IF EXISTS`)
3. **Data-safe** — does it preserve existing data? Column renames and type changes need explicit data migration
4. **Lock-safe** — does it avoid long table locks? Large tables need `CONCURRENTLY` for indexes, batched updates for data migrations
5. **Tested** — has it been run against a copy of production data?

## Failure Modes

| Failure | Cause | Mitigation |
|---|---|---|
| Migration conflicts | Two developers create migrations from the same parent | Rebase and regenerate — Alembic detects branch conflicts |
| Failed migration leaves dirty state | Migration crashes mid-execution | Wrap DDL in transactions (PostgreSQL supports transactional DDL) |
| Slow migration locks table | `ALTER TABLE` on a large table | Batch data changes, use `CONCURRENTLY` for indexes |
| Data loss on downgrade | `downgrade()` drops a column with data | Review downgrade paths — some migrations are intentionally irreversible |
| Env drift | Migration applied in dev but not staging | CI runs `alembic upgrade head` in every environment |
